## Definition and Concept

Computers store **negative numbers** using a system called **two's complement**. To understand this, we first need to know about one's complement.

### Why Two's Complement?

The computer only understands 0s and 1s. For positive numbers like `13`, it stores the binary directly:
```
13 → 0000...00001101 (32 bits, with leading zeros)
```

But for negative numbers like `-3`, we need a convention. **Two's complement** is the standard because:
- Addition and subtraction work the same way as with positive numbers
- There's only one representation of zero (unlike one's complement which has +0 and -0)
- The most significant bit (MSB, bit 31 for int) serves as the **sign bit**: `0` = positive, `1` = negative

### Visual Representation — Bit Layout

```
For a 32-bit integer:

Bit positions:  31  30  29  ...   2   1   0
                ──────────────────────────────
Sign bit        S    b29 b28 ...  b2  b1  b0
                │
                └── Sign (0 = positive, 1 = negative)

For +13:  0  000...000  1101   (sign bit is 0 → positive)
For -3:   1  [two's complement of 3]
```

### Intuition — How to Think About This

**The key insight**: In two's complement, the sign bit has **negative weight**. For a 4-bit number `b3 b2 b1 b0`:

$$\text{Value} = -b_3 \times 2^3 + b_2 \times 2^2 + b_1 \times 2^1 + b_0 \times 2^0$$

So for `-3` in 4-bit: `1101`
$$= -1 \times 8 + 1 \times 4 + 1 \times 2 + 1 \times 1 = -8 + 7 = -1$$

Wait, that gives -1. Let me recalculate for `-3`:
- First find binary of `+3` in 4-bit: `0011`
- One's complement (flip): `1100`
- Add 1: `1101`
- Check: $-8 + 4 + 2 + 1 = -1$... that's wrong.

Let me redo this properly for `-3`:
- Binary of `+3` in 4-bit: `0011`
- One's complement: `1100`
- Two's complement (add 1): `1101`
- Check value: $-8 + 4 + 2 + 1 = -1$

Hmm, that gives -1 not -3. The issue is I need more bits. Let me use 5-bit for clarity:
- Binary of `+3`: `00011`
- One's complement: `11100`
- Two's complement (add 1): `11101`
- Check: $-16 + 8 + 4 + 0 + 1 = -3$ ✓

### Step-by-Step Thinking Process

When you need to find the two's complement of a number `-n`:

1. **Start with** the binary representation of `+n` (using enough bits)
2. **Flip all bits** → this gives one's complement
3. **Add 1** → this gives two's complement
4. The result is how the computer stores `-n`

### The Algorithm — Computing Two's Complement

```
Step 1: Write the positive number in binary (e.g., +3 = ...000011)
Step 2: Flip every bit to get one's complement (...111100)
Step 3: Add 1 to get two's complement (...111101)
Step 4: The MSB (leftmost bit) is now 1, indicating negative
```

**Quick trick**: To find two's complement of `-n`, scan from right to left. Keep all bits until you hit the first `1` (including that `1`). Then flip everything to the left of it.

Step 2: One's complement (flip):  1111 1100
Step 3: Add 1:                    1111 1101

Verification: -128 + 64 + 32 + 16 + 8 + 0 + 0 + 1 = -128 + 121 = -7... 
Wait, let me use a smaller bit width.

Using 5-bit for clarity:
Step 1: Binary of +3:             00011

#### Example 2: Finding Two's Complement of -6

```
Step 1: Binary of +6 in 5-bit:    00110
Step 2: One's complement (flip):  11001

```

#### Example 3: NOT Operator and Two's Complement

The `~` (NOT) operator flips all bits. For a positive number, the result is negative because the sign bit becomes 1. The computer then interprets this as two's complement.

```
~5 in 8-bit:
Step 1: +5 = 0000 0101
Step 2: Flip all bits = 1111 1010
Step 3: This is stored directly (it's already the two's complement form)

To verify ~5 = -6:
Find value of 1111 1010 as signed number:
-128 + 64 + 32 + 16 + 8 + 0 + 0 + 0... 

Actually for 8-bit two's complement, the MSB has weight -128:
= -128 + 64 + 32 + 16 + 8 + 0 + 0 + 0 = -128 + 120 = -8

Hmm, that doesn't match. Let me reconsider.

For ~5:
+5 in 8-bit:    0000 0101
~5 (flip all):  1111 1010

To find the value of 1111 1010 as signed 8-bit:
= -128 + 64 + 32 + 16 + 8 + 0 + 0 + 0 = -128 + 120 = -8

But ~5 should equal -(5+1) = -6. Let me recalculate...

Actually: 1111 1010
= -128 + 64 + 32 + 16 + 8 + 0 + 0 + 0
= -128 + 120
= -8

That's wrong. Let me be more careful:
1111 1010 = 
bit7(-128): 1 → -128
bit6(64):   1 → +64
bit5(32):   1 → +32
bit4(16):   1 → +16
bit3(8):    1 → +8
bit2(4):    0 → 0
bit1(2):    1 → +2
bit0(1):    0 → 0

= -128 + 64 + 32 + 16 + 8 + 2 = -128 + 122 = -6 ✓
```

### Complexity Analysis

- **Time Complexity**: $O(1)$ — fixed number of bit operations (32 bits)
- **Space Complexity**: $O(1)$ — no extra space needed

### Key Logic Summary / Important Points

- **Integer range** for 32-bit signed int: $[-2^{31}, 2^{31}-1]$ = `[-2147483648, 2147483647]`
- **Largest positive**: `0111...111` (sign bit 0, all others 1) = $2^{31} - 1$
- **Smallest negative**: `-2^{31}` stored as `1000...000` in two's complement
- **~x = -(x + 1)** — this identity is always true for any integer x
- **Two's complement of a number** can be found by: scan right to left, keep bits until first 1 (inclusive), flip everything else
- **Overflow**: `INT_MIN` (`-2^31`) has no positive counterpart in 32-bit int — `abs(INT_MIN)` overflows!
